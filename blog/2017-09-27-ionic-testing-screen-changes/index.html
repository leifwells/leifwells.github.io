Testing in Ionic: Testing UI Changes

Sep 27, 2017

I've been thinking a lot about automated testing for Ionic projects (see Testing in Ionic: Configure Existing Projects for Testing and Testing in Ionic: Code Coverage). It is part of my development practice that has, until recently, been weak. So far this year, I've added unit testing to Ionic & Angular projects along with code coverage, the ability to review how well your unit tests are testing your code. I've also added end-to-end (E2E) testing, automated testing of applications. Eventually, I will be using continuous integration (CI), an integrated system that automates testing. Soon.

Lately, an important question has been bouncing in my mind: what happens when someone submits a pull request (PR) that inadvertently changes the UI in such a way that no one notices? Not the developer, not the reviewer, not the quality assurance developer. No one. It could be something simple, like an icon goes missing. Or CSS changes in one element effecting another element (an element the developer didn't touch)? A color change? A label change?

What if we could detect changes to screens and can review them before we merge a PR? Wouldn't that be nice?
